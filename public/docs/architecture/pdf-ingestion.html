    PDF-Ingestion – Architektur, Ablauf und Module¶

    Ziel¶

    • Klarer End-to-End-Flow von PDF/Worker-Callback zu Shadow-Twin (.md), Kapitel-/Frontmatter-Anreicherung und RAG-Ingestion.
    • Strikte Modularisierung: Orchestratoren rufen reine Unterfunktionen auf; Logs/Events bleiben konsistent für UI und Debugging.

    Phasen und Gates (Kurz)¶

    • Phase 1 extract_pdf: Worker liefert Text/Bilder; Shadow-Twin wird erzeugt.
    • Phase 2 transform_template: Template-LLM generiert/normalisiert Frontmatter, Kapitel/TOC ggf. via Analyzer repara3.
    • Phase 3 ingest_rag: Markdown aus Storage chunken/embedd3n; Chunks/Doc-Meta in Pinecone.
    • Gates: gateTransformTemplate (Skip, wenn Artefakt komplett), gateIngestRag (Skip, wenn Doc bereits Vektoren hat).

    Orchestratoren / Routen¶

    • app/api/external/jobs/[jobId]/route.ts (POST): Zentraler Orchestrator ffcr Worker-Callbacks.
    • app/api/external/jobs/[jobId]/retry/route.ts (POST): Template-Only/Retry nutzt dieselben Module und triggert internen Callback.

    Module & Kernfunktionen¶

    • Kontext & Sicherheit
    • src/lib/external-jobs/context.ts: readContext (Request/Params/Body/Job laden)
    • src/lib/external-jobs/auth.ts: authorizeCallback, guardProcessId
    • src/lib/external-jobs/policies.ts: readPhasesAndPolicies
    • src/lib/external-jobs/progress.ts: handleProgressIfAny

    • src/lib/external-jobs/provider.ts: buildProvider

    • Template/Analyse/Storage/Ingestion

    • src/lib/external-jobs/template-files.ts: pickTemplate (sucht/legt Template in /templates)
    • src/lib/external-jobs/template-decision.ts: decideTemplateRun
    • src/lib/external-jobs/template-run.ts: runTemplateTransform
    • src/lib/external-jobs/chapters.ts: analyzeAndMergeChapters
    • src/lib/external-jobs/storage.ts: saveMarkdown
    • src/lib/external-jobs/images.ts: maybeProcessImages
    • src/lib/external-jobs/ingest.ts: runIngestion

    • src/lib/external-jobs/complete.ts: setJobCompleted

    • Markdown Utils

    • src/lib/markdown/frontmatter.ts: parseFrontmatter, stripAllFrontmatter
    • src/lib/markdown/compose.ts: createMarkdownWithFrontmatter

    Ablauf (Mermaid)¶

    flowchart TD
  A[POST /api/external/jobs/[jobId]] --> B[readContext]
  B --> C[authorizeCallback]
  C --> D{handleProgressIfAny?}
  D -- yes --> Z[short-circuit progress]
  D -- no --> E[readPhasesAndPolicies]
  E --> F[buildProvider]
  F --> G[decideTemplateRun]
  G -- skip --> H[Frontmatter aus Body dcbernehmen]
  G -- run --> I[pickTemplate]
  I --> J[runTemplateTransform]
  J --> K[analyzeAndMergeChapters]
  K --> L[createMarkdownWithFrontmatter]
  L --> M[saveMarkdown]
  M --> N[maybeProcessImages]
  N --> O{ingest gate?}
  O -- run --> P[runIngestion]
  O -- skip --> Q[mark ingest skipped]
  P --> R[setJobCompleted]
  Q --> R
  R --> S[200 OK]

    Wichtige Logs/Events (Auszug)¶

    • Entscheidung/Analyse: template_decision, chapters_analyze_call, chapters_analyze_result
    • Persistenz: stored_local, stored_path, template_done
    • Ingestion: ingest_start, ingest_chunking_done, ingest_pinecone_upserted
    • Abschluss: completed

    Routen-Details (kurz)¶

    • route.ts (POST): 1) readContext → authorizeCallback → handleProgressIfAny 2) Policies/Provider laden → decideTemplateRun 3) Template-Fall: pickTemplate → runTemplateTransform → analyzeAndMergeChapters 4) Markdown bauen: createMarkdownWithFrontmatter → saveMarkdown 5) maybeProcessImages → Ingestion (runIngestion) → setJobCompleted

    • retry/route.ts (POST):

    • Template-Only Pfad: pickTemplate → runTemplateTransform → createMarkdownWithFrontmatter → interner Callback an route.ts (Phase 2/3 laufen zentral).

    Umgebungsvariablen (relevant)¶

    • SECRETARY_SERVICE_URL, SECRETARY_SERVICE_API_KEY
    • INTERNAL_TEST_TOKEN (interner Callback/Bypass)
    • PINECONE_API_KEY, NEXT_PUBLIC_APP_URL

    Tests (Stand)¶

    • Unit: template-decision/run, images, storage, ingest, complete.
    • Empfohlen: context/auth/policies, retry template-only Flow (internes Callback).

    Hinweise ffcr Programmierer¶

    • Module sind bewusst klein und rein (keine Nebenwirkungen audferhalb ihrer Aufgabe).
    • Orchestratoren halten den Kontrollfluss, Fehlerpfade sind in Guards/Module aufgeteilt.
    • Logs bleiben die prime4re Debugquelle; Events spiegeln UI-Statuswechsel.
